ve-core

verifiable encryption for key recovery.

hand someone an encrypted key share and a proof that it decrypts to exactly the share you committed to. they check it offline, with public data, before anyone reveals a secret.

what it is

recovery usually runs on trust. a server keeps a copy, or a backup sits in a drawer, and you find out whether it actually works the day you need it. that is a bad day to find out.

ve-core flips it. the party you seal a share to can prove, in zero knowledge, that the ciphertext decrypts to the share you committed to. the proof is public-coin: any verifier checks it with public data, no secret and no decryption. the holder of the recovery key decrypts later. nobody has to trust that the sealed share is good. they verify it.

a plaintext is a secp256k1 scalar. seal it to a recovery public key, get back a ciphertext carrying its own correctness proof.

two backends, one interface

ve-core is the seam: the plaintext, the binding context, and the backend trait family. pick a backend and the rest is the same.

  • cl-hsmq-encryption. verifiable encryption in the CL framework: the BICYCL construction with the BCL24 ciphertext-correctness proof, over imaginary-quadratic class groups. pure rust, no gmp, no gpl.
  • ec-segve. segmented verifiable encryption on native secp256k1. the share is split into limbs, each encrypted under curve elgamal, with base-b range proofs, an exact-integer carry chain, and a linking sigma, all under one fiat-shamir challenge. it speaks the same curve a hardware wallet already does, so the recovery key can be seed-rooted on the device.

what the proof buys

the proof is knowledge-sound. one accepting transcript extracts the whole witness, so a sealed package cannot decrypt to anything but the share it committed to. a malicious sealer cannot plant garbage that only fails the day you recover. that is the anti-ransom property, and it is checked at sealing time, recipient absent.

secrecy is the usual reduction: to anyone without the recovery key the package leaks nothing beyond the public commitment. and the accept path is bound by the type system. verify does not hand back an ok; it hands back the commitment it bound, and you accept only if that commitment equals the certified share. there is no shape of the code where you forget the check.

using it

let (pk, sk) = keygen()?;
let ct = pk.seal(&share, &ctx)?;          // ciphertext + correctness proof
pk.verify(&ct, &target, &ctx)?;           // proof holds AND commitment == target
let share = sk.open(&ct, &ctx)?;          // verify, then decrypt

pure rust. variable-time, for software adversaries observing messages and stored ciphertexts, not local side-channel attackers. heavily audited. not published yet: no crates.io, no public source.
