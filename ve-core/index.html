ve-core

verifiable encryption for key recovery.

prove what a ciphertext decrypts to without revealing it. here: that it holds the private key for a given public key, checkable by anyone from public data alone.

what it is

verifiable encryption lets you prove what a ciphertext decrypts to without revealing it, and without decrypting it. the proof travels with the ciphertext, and anyone checks it from public data.

in ve-core the claim is specific: that the ciphertext holds the private key for a given public key, the key that signs so signatures verify under it. so a holder of only the public key can confirm exactly what is sealed inside, in zero knowledge, and never see it.

the obvious use is recovery. recovery usually runs on trust: a server keeps a copy, a backup sits in a drawer, and you learn whether it works the day you need it. this lets you check up front instead. it is the difference between a spare key you assume fits and one you have already tried in the lock.

one key, or a package

the simplest use seals one secret to one recovery key. you check the ciphertext against a public key, and the proof guarantees the sealed secret is the private key for it: the exact key that signs under that public key. one ciphertext, one proof, one holder who can decrypt it.

or split the secret so no one holds it alone. each share is sealed to its own recovery key with its own proof, and the shares' public keys add up to your account's public key. that bundle is a package. check every share against its public key and you have proven the package restores your account key, with no single holder ever seeing the whole thing. a swapped, tampered, or junk share fails its check, long before you try to recover.

two backends, one interface

ve-core is the seam: the plaintext, the binding context, and the backend trait family. pick a backend and the rest is the same.

  • cl-hsmq-encryption. verifiable encryption in the CL framework: the BICYCL construction with the BCL24 ciphertext-correctness proof, over imaginary-quadratic class groups. pure rust, no gmp, no gpl.
  • ec-segve. segmented verifiable encryption on native secp256k1. the share is split into limbs, each encrypted under curve elgamal, with base-b range proofs, an exact-integer carry chain, and a linking sigma, all under one fiat-shamir challenge. it speaks the same curve a hardware wallet already does, so the recovery key can be seed-rooted on the device.

what makes it hold up

it cannot lie about what is inside. the proof is knowledge-sound: a single passing check pins down the secret, so the ciphertext cannot decrypt to anything but the private key for the public key you checked against. that is what stops someone sealing junk, or the wrong key, that only surfaces the day you try to recover. it is checked when the share is made, with the recipient absent.

it does not reveal the secret. to anyone without the recovery key, a sealed share leaks nothing beyond its public commitment, which was public to begin with.

you cannot forget to check. verify does not hand back a bare ok. it hands back the commitment it just proved, and you accept only when that commitment is the one you expected. the check is the return value, not a step you might skip.

using it

let (pk, sk) = keygen()?;
let ct = pk.seal(&share, &ctx)?;          // ciphertext + correctness proof
pk.verify(&ct, &target, &ctx)?;           // proof holds AND the secret is the private key for `target`
let share = sk.open(&ct, &ctx)?;          // verify, then decrypt

pure rust. variable-time, for software adversaries observing messages and stored ciphertexts, not local side-channel attackers. heavily audited. not published yet: no crates.io, no public source.
