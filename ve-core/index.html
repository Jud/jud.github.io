ve-core

verifiable encryption for key recovery.

hand someone an encrypted key share and a proof that it decrypts to exactly the share you committed to. they check it offline, with public data, before anyone reveals a secret.

what it is

recovery usually runs on trust. a server keeps a copy, or a backup sits in a drawer, and you learn whether it actually works the day you need it. bad day to find out.

ve-core lets you check first. you encrypt a secret to a recovery key and attach a short proof. anyone can read that proof and confirm one thing, in plain terms: this ciphertext really will decrypt to the value it claims. no decrypting it, no secret of yours, just public bytes. the holder of the recovery key opens it later, only if they ever have to.

so you never take a backup on faith. it is the difference between a spare key you assume fits and one you have already tried in the lock.

a sealed share, and a package

the basic unit is one sealed share: a single secret, encrypted to one recovery key, carrying its own proof. checking it tells you something exact. not that it is a valid encryption of something, but that it decrypts to the specific value you were expecting.

real recovery uses a package: your key is split into several shares, each sealed to a different holder, and the shares are arranged so their public commitments add back up to your key. check every sealed share against its commitment and you have proven the whole package restores your actual key. one swapped, tampered, or junk share is caught when the package is checked, not when you are trying to recover.

two backends, one interface

ve-core is the seam: the plaintext, the binding context, and the backend trait family. pick a backend and the rest is the same.

  • cl-hsmq-encryption. verifiable encryption in the CL framework: the BICYCL construction with the BCL24 ciphertext-correctness proof, over imaginary-quadratic class groups. pure rust, no gmp, no gpl.
  • ec-segve. segmented verifiable encryption on native secp256k1. the share is split into limbs, each encrypted under curve elgamal, with base-b range proofs, an exact-integer carry chain, and a linking sigma, all under one fiat-shamir challenge. it speaks the same curve a hardware wallet already does, so the recovery key can be seed-rooted on the device.

what makes it hold up

it cannot lie about what is inside. the proof is knowledge-sound: a single passing check pins down the whole secret, so a sealed share cannot decrypt to anything but the value it committed to. that is what stops someone sealing junk, or the wrong key, that only surfaces the day you try to recover. it is checked when the share is made, with the recipient absent.

it does not reveal the secret. to anyone without the recovery key, a sealed share leaks nothing beyond its public commitment, which was public to begin with.

you cannot forget to check. verify does not hand back a bare ok. it hands back the commitment it just proved, and you accept only when that commitment is the one you expected. the check is the return value, not a step you might skip.

using it

let (pk, sk) = keygen()?;
let ct = pk.seal(&share, &ctx)?;          // ciphertext + correctness proof
pk.verify(&ct, &target, &ctx)?;           // proof holds AND the value is exactly `target`
let share = sk.open(&ct, &ctx)?;          // verify, then decrypt

pure rust. variable-time, for software adversaries observing messages and stored ciphertexts, not local side-channel attackers. heavily audited. not published yet: no crates.io, no public source.
