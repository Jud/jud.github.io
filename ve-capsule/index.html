Encrypt one secp256k1 scalar to a fixed recipient, optionally gated behind a qualifying quorum of access keys.

Lifecycle

Four verbs, two phases

Encryption is one call. Decryption is consent-gated: verify hands you a capability token, and only that token can act.

01 · seal
seal
Lock a scalar to Y* = recipient + gates. Yields an opaque, immutable capsule.
02 · confirm
verify
Confirm the capsule against your authorization → a VerifiedCapsule token.
03 · gate
contribute
Each authorizer emits a gate-tagged Partial. Non-destructive.
04 · recover
unseal
The recipient gathers partials, buckets per gate, and recovers m.

// contribute & unseal exist only on the verified token — you cannot act on a capsule you haven't confirmed.

Surface

The core types

Concrete secp256k1 types throughout (the Context trait, Params, and the verb-traits round out the surface).

Capsule

The sealed, immutable ciphertext. Opaque outside the crate, and it never holds plaintext.

CapsuleBuilderborrows m

Collects the secret, recipient, context & gates; .seal() yields the capsule.

VerifiedCapsulecapability

A capsule you've confirmed. The only place contribute / unseal live.

Partial

One authorizer's gate-tagged decryption share + a batched DLEQ. A standalone, non-destructive artifact.

Case / VerifiedCase

The additive split of one secret to one recipient — verify checks Σ Mⱼ == M, then open every piece and sum to recover s.

PrivateKey

A zeroizing secret scalar — plaintext, key, or share. .public_key() derives its partner.

PublicKey

A nondegeneracy-checked secp256k1 point — the public partner of a PrivateKey.

Errornon_exhaustive

PointDecode · DegenerateInput · Verification.

In practice

One authorizer, one recipient

Two parties, both must act: the authorizer gates, the recipient opens. Neither can do it alone.

  recover.rs
use ve_capsule::{Capsule, PrivateKey, Context};

// every party derives its public key from its OWN secret — no co-generated pairs
let recipient_sk  = PrivateKey::from_secret(/* … */)?;
let recipient_pk  = recipient_sk.public_key();

let authorizer_sk = PrivateKey::from_secret(/* … */)?;
let authorizer_pk = authorizer_sk.public_key();

let m   = PrivateKey::from_secret(/* the scalar to protect */)?;
// `ctx` implements the `Context` trait — your integration binds each capsule
// to its own transcript (package id, epoch, purpose …).
let ctx = /* a value implementing Context */;

// SEAL — Y* = recipient + authorizer  (both must act)
let capsule = Capsule::builder(&m, &recipient_pk, &ctx)
    .access_key(&authorizer_pk)
    .seal()?;
let bytes = capsule.to_canonical_bytes();          // store / replicate opaquely

// … later, to recover …
let capsule = Capsule::from_canonical_bytes(&bytes)?;

// confirm against the authorization → a capability token (captures ctx)
let vc = capsule.verify(&m.public_key(), &recipient_pk, &[authorizer_pk], &ctx)?;

let partial   = vc.contribute(&authorizer_sk, &authorizer_pk)?;  // the authorizer's share
let recovered = vc.unseal(&recipient_sk, &[partial])?;           // == m
Composition

Splitting a secret

A Case is the additive split of one secret to one recipient — the DKG bundle. No party holds the whole secret; each dealer seals a single piece.

M  =  s·G  =  Σj Mj completeness is a sum of commitments — a missing or substituted piece breaks the equation

The recipient's whole secret is s = Σ sⱼ, with commitment M = s·G — a certified public output of the DKG. So Case::verify takes one tuple and checks Σ Mⱼ == M: completeness and correctness in a single equation — no roster and no signature needed. unseal opens every piece and sums them back to s.

  case.rs
impl Case {
    pub fn new(capsules: Vec<Capsule>) -> Result<Case, Error>;

    /// One tuple. Checks uniformity, each piece's π, and Σ Mⱼ == commitment.
    pub fn verify(&self, expected_commitment: &PublicKey, expected_recipient: &PublicKey,
                  expected_access_keys: &[PublicKey], ctx: &impl Context) -> Result<VerifiedCase, Error>;
}

impl VerifiedCase {
    pub fn contribute(&self, key: &PrivateKey, gate: &PublicKey) -> Result<Vec<Partial>, Error>;
    pub fn unseal(&self, recipient: &PrivateKey, partials: &[Partial]) -> Result<PrivateKey, Error>;
}
Guarantees

Crate invariants

Cryptographic, not optional checks.

I1

Recipient fixed at seal

The recipient is baked into Y* at seal. A quorum consents to opening — it can never choose or redirect who receives the secret.

I3

Opaque blob

Outside the crate a capsule is a storage-only blob. Nothing is surfaced; even verify only confirms against values you supply.

I4

Sealed by construction

A Capsule can't exist unsealed — no public constructor, no derived deserialize. Every value is the output of a seal.

Threat model

What holds, precisely

  • ∑
    All-or-nothing, strict AND. Partials are bucketed per gate and checked Σ wᵢ·G == Y_accessₖ per group — every group must independently reach quorum. One short, and the ciphertext is as inert as untouched (computationally, under DDH).
  • #
    The exact gate set is pinned. Seal binds a canonical gate commitment g*, not just the sum — so a different key list with the same sum can't be substituted in.
  • ↻
    No accepted replay. Each Partial's DLEQ binds the capsule core-hash, gate tag & context — it can't be reused against a different capsule, gate, or context.
  • ✓
    Consent-gated. contribute and unseal exist only on a VerifiedCapsule, closing the static-DH oracle: you never act on a capsule you haven't confirmed.
