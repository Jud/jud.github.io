Seal

Lock a key to a recipient

Seal a private key — or any secp256k1 scalar — to a recipient. What comes back is a capsule: opaque bytes that give nothing away. Store it, copy it, hand it around. It stays shut until the recipient opens it.

let capsule = Capsule::builder(&secret, &recipient, &ctx).seal()?;
let bytes   = capsule.to_canonical_bytes();   // just bytes — keep them anywhere

The recipient is a public key; sealing needs only that, opening needs the private key behind it. ctx binds the capsule to your application, so it can't be lifted and replayed somewhere else.

Verify

Prove the contents, reveal nothing

A plain ciphertext is a promise you can't check until you open it. A capsule carries a short, non-interactive proof that anyone can verify, against public data alone — so a backup can be audited the moment it's made, not discovered to be empty or wrong years later when you finally reach for it.

// anyone holding the capsule and the public key can run this — no secret needed
capsule.verify_ungated(&public_key, &recipient, &ctx)?;
// proven: this capsule holds the private key for `public_key`. Nothing opened.
Lifecycle

Four verbs, two phases

Encryption is one call. Decryption is consent-gated: verify hands back a capability token, and only that token can act — you never operate on a capsule you haven't confirmed.

01
seal
Lock a scalar to recipient + gates → an opaque, immutable capsule.
02
verify
Confirm it against your authorization → a VerifiedCapsule token.
03
contribute
Each authorizer emits a gate-tagged Partial — portable across devices.
04
unseal
The recipient gathers partials, buckets per gate, recovers the secret.
Compact recovery

A whole backup, tag-sized

The proof is for the auditor — recovery never re-runs it. Once a capsule is verified you can strip the proof and keep the small opening core. For a distributed key you can go further: because the pieces share one recovery key and EC-ElGamal is additively homomorphic, an entire split — any number of helpers — collapses into one synthetic capsule, the homomorphic aggregate.

≈149 KB × helpers → 789 bytes The entire aggregate, serialized — and constant: the same size whether two helpers sealed the split or two hundred. Small enough for an NFC tag, a QR code, or a slip of paper. 
// verify the bundle once (every piece present, Σ Mⱼ == M), then compact it
let aggregate = verified.aggregate()?;            // H pieces → one synthetic capsule
let bytes     = aggregate.to_canonical_bytes();   // 789 B, independent of H

// …later, from the tag…
let recovered = AggregateCase::from_canonical_bytes(&bytes)?
    .bind(&whole_key, &recipient, &[], &ctx)?       // re-pin to the certified target
    .unseal(&recipient_key, &[])?;                  // back to s = Σ sⱼ
Two trust models

Recover alone, or only by quorum

A compact core has dropped its proof — so something has to vouch for it. Two ways, and you pick the one your threat model wants.

Recipient-only self-securing

Nothing extra to keep. Recovery re-anchors on the certified target: the reconstructed scalar's commitment must equal M, so a tampered or substituted core fails closed. The recipient opens it with their key alone — the air-gapped backup.

Signature-backed open set

A signature takes the place of the proof: a quorum signs the capsule's canonical statement, promoting the core to a contributable token — "require another group to authorize release." The crate verifies BIP-340 Schnorr today (FROST(secp256k1)-TR among them); the backing is an open set, with room for other schemes.

Why a signature, and not just any bytes? A stripped core carries no proof — so on its own an authorizer can't tell a genuine capsule from one an attacker fabricated to harvest a partial decryption. The signature vouches that the core is real before anyone contributes, so an authorizer can't be duped into opening an attacker's chosen ciphertext.

Compose

Split a key into a Case

When a distributed key generation produces a key as shares that add up to the whole, seal each share and bundle the capsules into a Case. The recipient's secret is s = Σ sⱼ with commitment M = s·G — a certified output of the DKG — so verifying the Case checks Σ Mⱼ == M: completeness and correctness, in a single equation. No roster, no signature. An authorizer can itself be a multi-party key no single member holds.

let case     = Case::new(capsules)?;
let verified = case.verify(&whole_key, &recipient, &[], &ctx)?;
// every share present, and together they reconstitute the whole key
Why secp256k1

One curve, end to end

The recipient, the gates, the recovered secret, and the optional quorum signature all live on secp256k1. Many verifiable-encryption constructions live elsewhere — a class group, a pairing-friendly curve, an RSA modulus — and ask your key material to take a detour through that world. ve-capsule stays native.

That alignment is the point: a secp256k1-only secure element can sole-custody the recovery key, and the same signer that already protects your wallet can authorize a recovery — no second curve, no class group, nothing extra to port to hardware or to trust.

Under the hood

ve-capsule is segmented EC-ElGamal verifiable encryption on secp256k1, with per-segment range proofs and a batched DLEQ, made non-interactive by Fiat–Shamir. Every capsule binds to exactly its participants and its context, checkable by anyone without the secret. The surface is blob-in / blob-out: #![forbid(unsafe_code)], secrets zeroize on drop, and the proof algebra stays crate-private so a caller can't reach past the consent-gated API. Sealing and unsealing aren't constant-time — run them where the secret already lives.
