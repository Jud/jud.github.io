ve-capsule · verifiable encryption on secp256k1

Encryptedkeys. Publicchecks. Privaterecovery.

ve-capsule turns a secp256k1 secret into a capsule. Anyone can check it opens to the expected public key. Only the recipient, plus any required approvers, can recover it.

secp256k1 · segmented EC-ElGamal · Bulletproofs++ · compact recovery payloads · no unsafe
01 · the sealer

Creates the capsule from the secret and recipient public key.

02 · the verifier

Checks the capsule without learning the secret.

03 · the unsealer

Uses the recipient secret, and any gate contributions, to recover it.

The split

Storage should not require trust.

With normal encryption, a stored blob is opaque. The storage layer cannot prove it contains the right secret. ve-capsule adds that proof without giving storage the key.

01The sealer

Needs the secret, the recipient public key, and context bytes. The result is canonical bytes you can store anywhere.

let capsule = Capsule::builder(&secret, &recipient, &ctx).seal()?;
let bytes   = capsule.to_canonical_bytes();   // opaque; store them anywhere

02The verifier

Needs the capsule and public keys. The proof shows the capsule opens to public_key. Nothing used for verification can open it.

// no secret anywhere in this call
capsule.verify_ungated(&public_key, &recipient, &ctx)?;
// proven: the right key is inside. Still sealed.

03The unsealer

Needs the recipient secret. Verify the capsule first, then recover the sealed scalar.

let verified = Capsule::from_canonical_bytes(&bytes)?
    .verify_ungated(&public_key, &recipient, &ctx)?;
let secret   = verified.unseal(&recipient_secret, &[])?;

ctx binds the capsule to your application, so the same bytes cannot be replayed in another context.

What it does

Inputs and outputs.

Callers pass in scalar material, public keys, and context bytes. ve-capsule returns canonical artifacts for storage, verification, signing, and recovery.

  • Seal

    Encrypt one scalar to a recipient public key. Optional gates add approval requirements.

  • Verify

    Check the capsule opens to the expected public key without learning the scalar.

  • Recover

    Open with the recipient secret and any required approvals. Reject if the result does not match the expected public key.

Consent

Add approvals when one key is not enough.

A gate is an extra public key that must contribute before the capsule opens. The recipient key can open alone only when there are no gates. A gate can also be a threshold key.

let capsule = Capsule::builder(&secret, &recipient, &ctx)
    .access_keys(&quorum)
    .seal()?;   // now unsealing requires their contributions too

A verified capsule can be stripped down to a compact core. A quorum signature over that core can stand in for the original proof, so approvers do not help open fabricated ciphertexts.

Compact

Split the secret. Keep recovery small.

If a scalar is stored as additive pieces, seal each piece and bundle the capsules into a Case. One check, Σ Mⱼ == M, proves the pieces add up to the expected public key. For recovery, each producer also emits a 65-byte hint. The hints form a compact recovery payload. If you need interpolation weights, apply them before sealing.

≈5.4 KB / proof → 65 bytes a piece Recipient-only: one ECDH, one hash, and one subtraction per piece. The recovered sum must match the expected public key. 
let rctx = RecoveryContext {
    certified_target: &expected_public_key,
    ctx: &context_bytes,
    epoch,
};

// producers seal hints beside their capsules; the payload carries no proofs
let hint = seal_recovery_hint(&contribution, &recipient, &rctx, idx, &mut rng)?;
let payload_bytes =
    assemble_recipient_recovery_payload(&expected_public_key, &hints)?;   // no proof bytes
One curve

Native to secp256k1.

The recipient, gates, recovered secret, and quorum signature all live on secp256k1. No class group, pairing curve, or RSA modulus. It reuses the keys and signers already used by Bitcoin, FROST, and Taproot systems.

Why not just BP++?

Bulletproofs are only one part.

A Bulletproof can prove committed numbers are in range. It does not define who can open the ciphertext, how recovery is checked, how approvers contribute, or how a large proof becomes a compact recovery payload.

ve-capsule uses Bulletproofs++ for the range proof. Around it, the crate provides encryption, context binding, recipient opening, consent gates, quorum signatures for stripped cores, Cases for additive pieces, and recovery hints.

Under the hood

Segmented EC-ElGamal, one transcript.

The scalar is encrypted in small segments. One aggregated Bulletproofs++ proof covers the segments and carries, with a linking sigma and batched DLEQ on the same Fiat-Shamir transcript. Verification is a handful of multiscalar equations, and there is no trusted setup. Each capsule binds to its keys and context. Secrets zeroize on drop. The proof algebra stays crate-private. Sealing and unsealing are not constant-time; run them where the secret already lives.
